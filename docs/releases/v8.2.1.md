# NOOP v8.2.1: the continuation release

This is the first release from the community-maintained continuation at
[github.com/neekolascmd/noop](https://github.com/neekolascmd/noop). It preserves the local-first app
and existing data model while establishing a reviewable release process around the abandoned
upstream project. NOOP remains independent, experimental, and unaffiliated with WHOOP.

## What changed

- **Backup restores fail safely.** The Apple database is integrity-checked before it replaces the
  live store, verified again after the swap, and rolled back if either check fails. Regression tests
  cover synchronous and asynchronous restore paths, malformed archives, migration, and rollback.
- **Required checks always report.** Apple, Android, and shared Swift-package workflows now produce
  stable required-check names even when a change does not touch that platform.
- **Release artifacts are reproducible and inspectable.** The manual and tag workflows build the
  macOS zip, unsigned iOS IPA, and Android APK from the reviewed commit. The publishing job verifies
  archive structure and embedded Apple versions before attaching SHA-256 checksums.
- **Analytics parity is pinned across Swift and Kotlin.** Shared golden fixtures cover daily metrics,
  exercise and sleep analytics, wearable imports, and invalid-input behavior on both platforms.
- **Hardware claims are auditable.** The new support matrix distinguishes physical tests from
  protocol-level evidence and experimental support. It also records firmware sensitivity and links
  a structured verification report.
- **Project stewardship is explicit.** Repository links, contribution guidance, issue routing, and
  historical documentation now point to the maintained fork without rewriting original credit.

No new device capability is claimed by this patch. WHOOP 4.0 remains the supported path. WHOOP 5.0,
WHOOP MG, and direct Oura access remain experimental; see
[hardware support matrix](https://github.com/neekolascmd/noop/blob/v8.2.1/docs/HARDWARE_SUPPORT.md)
for the evidence behind each claim.

## Downloads and platform limitations

All artifacts are built by GitHub Actions from tag `v8.2.1`. Verify the download using the attached
`SHA256SUMS.txt` before installing.

- **macOS — `NOOP-v8.2.1-macos.zip`.** Universal Apple Silicon and Intel application. The bundle is
  ad-hoc signed and **not Apple-notarized** because the project has no paid Developer ID. Gatekeeper
  may require Control-click → Open on first launch; see the README before changing quarantine flags.
- **iOS — `NOOP-v8.2.1-ios.ipa`.** The IPA is unsigned and contains no embedded Watch app. Sign it
  with your own identity through AltStore, SideStore, or Xcode. Free Apple IDs normally require a
  refresh every seven days, and HealthKit, Live Activities, widgets, and background behavior can be
  limited by the resulting provisioning profile.
- **Android — `NOOP-v8.2.1.apk`.** Sideload the full flavor on Android 8 or newer. It is not a Play
  Store build and Play Protect may warn. It is signed with the continuation repository's stable
  release key, which the release workflow verifies. Certificate SHA-256:
  `15:0F:92:20:F6:58:1E:01:91:84:2C:B2:40:9B:34:50:EC:1D:DE:B6:36:64:46:A7:8B:15:89:56:51:FB:34:10`.
  A differently signed historical build must be backed up and uninstalled once; continuation
  releases from v8.2.1 onward should upgrade in place.
- **Unsupported platforms.** There is no supported Windows or Linux application in this release.
  Windows remains roadmap work; Linux tooling is research-oriented and is not a consumer app.

## Stewardship and known issues

The original author and prior contributors retain their credit in the history. This fork is a
community continuation, not an official successor endorsed by the former maintainer. Reports should
include the exact version, host OS, hardware model, firmware, and reproducible evidence. The
[support matrix](https://github.com/neekolascmd/noop/blob/v8.2.1/docs/HARDWARE_SUPPORT.md) and
[hardware verification form](https://github.com/neekolascmd/noop/issues/new?template=hardware-verification.yml)
describe what is verified today and what still needs hardware testing.
